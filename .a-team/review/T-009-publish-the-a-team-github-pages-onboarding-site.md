# T-009 — Publish the A-Team GitHub Pages onboarding site

## Outcome

A tech lead or technical PM landing on the public A-Team site understands why chat-only agent execution becomes unmanageable at AI scale, sees how A-Team converts intent into human-sized executable work packages, and follows T-008's verified path to a first valid ticket.

## User goal

Understand the human-oversight problem, see A-Team's control model, and reach the first valid ticket through one trusted path.

## Entry point

`https://arpadtamasi.github.io/a-team/`, the README link to that URL, or the repository link from the page.

## Default state

Semantic HTML renders the hero, navigation, and both CTAs before JavaScript executes. The six content units appear in this order: hero, problem comparison, workflow, layer comparison, quickstart, trust/footer.

## Loading state

There is no content loading state. Local/system font fallbacks render immediately; no network font is required.

## Empty state

Not applicable because all content is built in. Missing required content is a build failure.

## Error state

Broken build, assets, fragments, or links fail CI. If runtime JavaScript fails, the semantic HTML remains readable and all native anchors work.

## Success state

The quickstart and complete valid-ticket example are visible, and the canonical live URL returns HTTP 2xx under the `/a-team/` base path.

## Disabled state

No disabled controls exist in V1.

## Responsive behaviour

At 375×812 and 1440×900 there is no page-level horizontal overflow. Navigation wraps rather than hiding behind a menu. Workflow order and comparison headers remain understandable.

## Keyboard and focus behaviour

Native anchor order follows the DOM. Every link and CTA has a visible focus indicator and activates with Enter. There is no focus trap, custom shortcut, or hamburger menu.

## Accessibility expectations

Semantic headings and names are required; text and controls meet WCAG AA contrast; reduced-motion mode disables nonessential motion; `@axe-core/playwright` reports zero serious or critical findings.

## Visual reference

Use the A-Team tokens by value without importing dashboard components: paper `#f1eddf`, light paper `#fbf8ee`, ink `#17140f`, orange `#ff5a1f`, acid `#caff3d`, blue `#2674ff`. Use the existing serif body and condensed display system-font stacks. The layout is editorial and card-light, with strong rules, large headings, a two-column problem comparison, five-step workflow, one comparison table, and one code block. The approved design document is `/Users/rp/.gstack/projects/arpadtamasi-a-team/rp-main-design-20260722-091654.md`; its `/tmp` screenshot reference is non-normative.

## Actors

Contributor, fork contributor, maintainer, GitHub Actions, GitHub Pages, and the target tech lead/technical PM.

## Initial state

T-008 is done and exposes a frozen, measured public install path. No public A-Team Pages site or Pages workflow exists.

## States

Install dependencies → site build → local artifact server → Playwright/axe/link/network checks → artifact upload → deploy → live URL verification → done.

## Transitions

Every trigger enters the focused site check. Pull requests stop after artifact checks. Only `arpadtamasi/a-team` on `refs/heads/main` may upload and deploy. Deploy success enters live verification; a 2xx canonical URL completes the workflow. Failed gates cannot be skipped.

## Triggers

Pull requests build without deployment, `main` pushes build and deploy, and `workflow_dispatch` may deploy only from `main`.

## Permissions

Build jobs are read-only. The deploy job receives only `contents: read`, `pages: write`, and `id-token: write`, gated by the repository/ref condition and protected `github-pages` environment. Fork PRs receive no deployment authority.

## Error paths

Build or pre-deploy artifact-check failure produces no upload or deploy. Missing artifacts block deploy. A failed post-deploy canary detects platform/configuration failure; recovery is a corrected `main` redeploy.

## Cancellation path

The `pages` concurrency group uses `cancel-in-progress: true`. Cancellation before deploy leaves the live site untouched; cancellation after deploy is treated as failed live verification.

## Retry and duplicate-action behaviour

Rerunning the same commit is safe. Artifact and deployment creation do not mutate repository state. Equivalent bytes may be redeployed with a new environment audit record.

## Audit and notification expectations

The Actions run and `github-pages` environment record the commit, actor, artifact, and URL. Default GitHub failure notification is sufficient for V1.

## Scope

- Create a separate static Vite application in `site/`, with `site-dist/`, `dev:site`, and `build:site`, independent of the local `/api/*` dashboard runtime.
- Use production Vite base `/a-team/`; local development serves the same entry at `/`.
- Implement the six approved content units and exact positioning from the approved design doc.
- Provide navigation: `Why A-Team` → `#why`, `How it works` → `#how`, `Install` → `#install`, and `GitHub` → `https://github.com/arpadtamasi/a-team`.
- Show the T-008 verified commands, supported-host boundary, measured duration, and a complete minimal `.a-team/backlog/T-001-first-controlled-change.md` example.
- Add focused PR checks and main-only Pages deployment with pre-deploy artifact tests.
- After the first successful deployment, link README to `https://arpadtamasi.github.io/a-team/` while retaining its text quickstart.
- Cover mobile/desktop layout, keyboard focus, reduced motion, accessibility, base-path behavior, link integrity, and no-JavaScript readability.

## Non-goals

- Hosting the local dashboard or exposing a real workspace.
- Authentication, backend APIs, analytics, CMS, blog, search, versioned docs, custom domain, or translation.
- Redesigning the CLI, ticket lifecycle, skills, or local UI.
- Claiming Jira/Linear, chats, or orchestrators are obsolete.

## Acceptance

- A clean checkout succeeds with the documented Node version, locked dependencies, `build:site`, and focused site tests.
- The built page loads under `/a-team/`; first-party assets use the base path, fragments target unique IDs, and external repository links are absolute HTTPS URLs.
- The DOM contains the six units in order and the required hero: `AI can finish work too large for you to see.`
- Primary CTA `Create your first ticket` targets `#install`; secondary CTA `See the workflow` targets `#how`.
- The problem heading states `The agent is no longer the bottleneck. Human oversight is.` and the workflow heading states `Keep AI-sized output inside human-sized work packages.`
- The comparison contains exactly Agent chat, Jira/Linear, Agent orchestrator, and A-Team rows across intent, execution-control, and completion-verification dimensions, framed as different jobs rather than replacement.
- Quickstart uses the exact T-008 public package/skills path, selectable command text without Clipboard API dependency, and ends in a complete minimal ticket plus successful validation result.
- The site makes no `/api/*` request and requires no backend, login, database, or hosted service.
- Playwright at 375×812 and 1440×900 finds no page-level overflow; axe reports zero serious/critical findings; keyboard and reduced-motion checks pass.
- Fork PRs run `site-build` without deployment credentials; only successful `main` runs deploy through `github-pages`.
- `site-dist/` is absent from `npm pack --json`; existing CLI commands, dashboard behavior, typecheck, build, and tests are not regressed.
- README links to the verified live URL and retains T-008's text quickstart.
- Before copy freeze, one tech lead or technical PM who uses coding agents sees only the hero, problem comparison, and workflow. Their unprompted answer to `What does this tool control, and what would you try next?` mentions scope, decomposition, execution control, or evidence and identifies the first-ticket quickstart as the next action; otherwise revise and repeat once.

- Run `npm ci`, `build:site`, site Playwright/axe/link/network checks, existing typecheck, aggregate build, and full tests on `main`.
- Serve `site-dist/` beneath a local `/a-team/` prefix and assert all first-party responses are 2xx, every fragment resolves, external links are well formed, and no `/api/*` request occurs.
- Capture 1440×900 and 375×812 screenshots covering hero, workflow, comparison, and quickstart.
- Prove branch and fork PRs stop before deployment; then verify the `main` environment record and canonical live URL.
- Repeat T-008's exact-version clean-install canary during site review.
- Attach the comprehension participant's role, exact unprompted answer, date, and tested copy revision without their name/employer unless they consent.

## Constraints

Use Vite, GitHub Pages, GitHub Actions, the repository subpath, system fonts, minimal runtime JavaScript, and the approved content/visual contract. Do not couple the site to local application components or APIs.

## Open decisions

None.
